# 美食地图助手 PRD

## 1. 文档信息

- 产品名称：美食地图助手
- 产品形态：微信小程序
- 版本范围：V1
- 使用对象：个人用户，自用为主
- 文档目标：用于确认需求、指导设计与开发实施

## 2. 项目背景

用户会在 B 站、小红书等内容平台看到多家餐饮门店推荐，但一个视频或截图里通常同时包含多家店。用户如果逐个去地图里搜索门店、查看评分、判断位置分布，会消耗大量时间。

本产品的目标是把这段低效流程压缩为一次导入和一次筛选：

1. 批量导入店名文本或上传截图进行 OCR 识别。
2. 自动查询每家门店的核心信息。
3. 在导入前让用户快速筛掉不想收藏的商家。
4. 导入后直接在地图上查看所有已收藏商家的空间分布。

## 3. 产品目标

### 3.1 核心目标

- 让用户在一次操作中导入 1 至多家门店，而不是逐个手动搜索。
- 在导入前展示足够做判断的关键信息：评分、人均、地址、匹配状态。
- 导入后通过地图分布帮助用户识别商家聚集区域，而不是靠文字地址脑补位置关系。

### 3.2 成功标准

- 用户可在 1 次导入流程中完成多家门店收藏。
- 用户在导入前可手动取消部分商家，避免脏数据进入收藏库。
- 收藏后的地图页可以直观看到门店分布，并支持点击查看详情。

## 4. 目标用户

### 4.1 用户画像

- 经常刷 B 站、小红书、抖音等内容平台种草餐饮店的个人用户。
- 主要需求不是写点评，而是先快速收藏、整理与看地图分布。
- 对效率敏感，希望减少重复搜索和重复比对。

### 4.2 使用场景

- 从视频简介、评论区或笔记正文中复制多家店名进行导入。
- 上传 1 到 3 张截图，让系统识别其中的店名并批量搜索。
- 在导入结果页中取消不想去的商家。
- 在收藏地图页中观察哪些店集中在同一商圈或相邻区域。

## 5. 版本范围

### 5.1 V1 包含范围

## 6. 核心功能方案

### 6.1 功能一：批量导入

支持两种导入方式：

1. 粘贴多行文本
2. 上传截图 OCR

#### 6.1.1 文本导入规则

- 用户可粘贴任意文本内容。
- 系统优先按换行拆分，其次兼容顿号、逗号、分号、数字序号等常见分隔形式。
- 系统自动去掉空行、重复店名、明显非店名干扰词。
- 清洗后生成待搜索门店列表。

#### 6.1.2 图片 OCR 导入规则

- 最多上传 3 张图片。
- 支持连续上传多张截图。
- 所有图片 OCR 结果汇总后统一清洗、去重。
- OCR 输出结果进入与文本导入相同的后续搜索流程。
- 小程序端必须展示 OCR 处理中状态，避免用户误以为卡死。
- 1 到 3 张图片的 OCR 识别目标时间控制在 8 秒内。
- 超过 8 秒仍未完成时，界面持续提示“仍在识别中”。
- 超过 15 秒仍未完成时，直接提示识别失败并允许用户重试。

#### 6.1.3 清洗规则

- 去掉序号，如 `1.`、`01`、`No.1`
- 去掉明显无意义短词，如 `必吃`、`推荐`、`人均`
- 合并重复店名
- 保留原始识别文本，便于后续排查

### 6.2 功能二：门店自动识别

每个店名单独查询一次高德 POI 数据。

#### 6.2.1 查询策略

- 每个店名调用一次关键词搜索接口
- 优先要求用户选择目标城市，提升搜索准确率
- 查询时启用扩展字段，尽量拿到评分、人均、特色菜等信息
- 对于同名门店较多的情况，默认取最佳结果，并预留手动切换候选结果能力

#### 6.2.2 查询结果目标字段

- 门店名称
- 评分
- 特色菜标签
- 人均价格
- 经纬度
- 详细地址
- 商圈
- 高德 POI ID

### 6.3 功能三：导入前预览与筛选

系统识别完成后，进入导入预览页。

#### 6.3.1 卡片展示字段

- 门店名称
- 评分
- 人均价格
- 地址简述
- 匹配状态
- 选中状态
- 菜系 / 品类标签（若有）

#### 6.3.2 交互规则

- 列表默认全选
- 用户可逐条取消勾选
- 用户可批量全选 / 全不选
- 用户点击“确定导入”后，仅导入当前选中的门店
- 可导入门店必须优先展示在列表上方，弱匹配 / 不建议导入项必须沉到底部
- 在“全部”模式下，导入预览页应拆分为“推荐导入”和“暂不建议导入”两个分区
- 评分文案必须明确表达为“★ 4.4”或“4.4星”，禁止仅展示裸数字
- 不同评分等级必须使用不同颜色，同时保持同一暖色体系，避免与页面整体风格冲突
- 若没有菜系 / 品类标签，默认隐藏该字段，不展示“暂无标签”类占位文案

### 6.4 功能四：地图收藏展示

导入成功后，用户可在地图页查看所有已收藏商家。

#### 6.4.1 地图页目标

- 让用户看见门店分布，而不是只看列表
- 让用户直观识别商家是否集中在某一个区域
- 支持点击地图点位查看门店详情

#### 6.4.2 地图页展示内容

- 所有已收藏门店标点
- 当前视野自动包含所有点位
- 门店基础信息弹层或底部卡片
- 收藏总数
- 当前选中门店的取消收藏按钮
- 当前选中门店展开后的附近商家列表

### 6.5 功能五：收藏列表

地图页之外，需要一个列表页做补充浏览。

列表页展示：

- 门店名称
- 评分
- 3 个特色菜
- 人均价格
- 地址
- 导入时间

列表页支持：

- 查看详情
- 删除收藏
- 从列表跳转地图定位

### 6.6 功能六：地图城市筛选与聚焦（V1.1 优化）

#### 6.6.1 动态城市列表

地图页顶部城市筛选器的城市列表应该是动态的：

- **当前行为**：显示预设的固定城市列表（北京、上海、广州、深圳...）
- **期望行为**：只显示收藏夹中实际存在的城市，未收藏的城市不显示
- **实现方式**：
  - 从收藏数据中提取所有城市的 adcode
  - 动态生成城市列表，添加"全部"选项
  - 当用户有多个城市的收藏时，只显示有数据的城市

#### 6.6.2 城市选择后地图聚焦

当用户切换城市筛选器时：

- **当前行为**：仅过滤列表数据，地图视野不变
- **期望行为**：切换城市后，地图自动缩放并居中到该城市的门店位置
- **实现方式**：
  - 筛选后取该城市所有门店的经纬度范围
  - 使用 `includePoints` + `mapCtx.moveToLocation()` 或调整 `latitude/longitude` 实现视野聚焦
  - 若城市只有一家店，缩放到适中比例尺（建议 scale=14）
  - 若城市有多家店，计算所有门店的边界并调整视野包含所有点

#### 6.6.3 默认展示第一家门面

地图页打开时的默认状态：

- **当前行为**：打开地图时可能显示空状态（无选中门店）
- **期望行为**：打开地图时默认选中并展示收藏夹中的第一家门店
- **实现方式**：
  - 在 `onShow` 时，若无指定聚焦门店，自动选择列表第一个门店
  - 地图视野自动移动到该门店位置
  - 底部卡片显示该门店信息

#### 6.6.4 导入完成后的地图默认聚焦

当用户刚完成一轮导入并跳转到地图页时：

- **期望行为**：优先聚焦于刚刚导入的那一批门店，而不是直接回到“全部收藏”的全局视角
- **推荐规则**：
  - 若刚导入的门店均属于同一城市，地图默认切换到该城市范围
  - 默认选中“刚导入批次中评分最高的门店”作为当前门店
  - 底部卡片展示该门店信息，用户可以继续上滑查看其附近门店
- **目标价值**：
  - 避免用户导入后立刻迷失在全量收藏地图中
  - 让用户一进入地图页就知道刚刚导入的是哪个城市、哪家最值得先看

#### 6.6.5 "管理清单"按钮功能

地图页的"管理清单"按钮应该正确跳转到列表页：

- **实现方式**：使用 `wx.navigateTo` 跳转到 `/pages/list/index`（因为 list 不是 tabBar 页面）

#### 6.6.6 地图底部可拖动面板（V1.1 核心交互优化）

##### 设计目标

帮助用户快速发现聚集的商家，方便安排同一区域的用餐行程。

##### 面板布局

面板始终同时显示两部分：
1. **顶部区域**：当前选中的商家卡片
2. **下方区域**：附近商家列表（展开时显示）

##### 交互设计

**收起状态**：
- 只显示顶部商家卡片
- 卡片显示：店名、评分、标签、人均、地址
- 底部有"↑ 上拉查看附近商家"提示
- 点击卡片可触发导航

**展开状态**：
- 向上拖动面板或点击拖动指示器展开
- 显示顶部商家卡片 + 下方附近商家列表
- 附近商家按距离排序，距离越近越靠前
- 显示每个商家的店名、评分、距离、标签
- 展开后附近商家列表必须真实渲染可见，不能只显示标题而无列表内容

**点击附近商家**：
- 地图自动平移聚焦到该商家
- 顶部卡片更新为被点击的商家
- 下方列表更新为该商家的附近商家
- 面板保持展开状态

**无选中商家时**：
- 面板收起，显示空状态提示
- 提示"点击地图上的标记选择商家"

##### 附近商家定义

"附近"的计算逻辑：
- 在当前筛选结果（城市+关键词）内
- 按距离排序，取距离最近的 N 家（如 10 家）
- 距离计算使用 Haversine 公式

##### UI 样式

- 面板背景：白色，带圆角（顶部）
- 拖动指示器：灰色圆角条，居中
- 阴影：底部面板上方有轻微阴影
- 动画：拖动时有平滑过渡效果
- 附近商家列表项：紧凑布局，显示店名+评分+距离+标签
- 当前商家卡片中的评分颜色需要与全局评分色阶一致，不使用与页面整体风格冲突的高饱和金色

## 8. 关键业务流程

### 8.1 文本导入流程

1. 用户进入首页。
2. 粘贴店名文本。
3. 选择目标城市。
4. 点击开始识别。
5. 系统清洗文本并拆分店名。
6. 系统逐条调用高德接口查询门店。
7. 系统生成导入预览列表。
8. 用户取消部分商家勾选。
9. 用户点击确定导入。
10. 系统保存收藏并跳转地图页。
11. 地图页默认聚焦到本次导入城市，并优先选中本批次评分最高的门店。

### 8.2 OCR 导入流程

1. 用户进入首页。
2. 上传 1 到 3 张截图。
3. 选择目标城市。
4. 点击开始识别。
5. 系统调用 OCR 识别文本。
6. 系统对 OCR 文本做清洗、去重、拆分。
7. 系统逐条调用高德接口查询门店。
8. 系统生成导入预览列表。
9. 用户取消部分商家勾选。
10. 用户点击确定导入。
11. 系统保存收藏并跳转地图页。
12. 地图页默认聚焦到本次导入城市，并优先选中本批次评分最高的门店。

## 10. 外部能力接入方案

### 10.1 高德 POI 数据源

#### 10.1.1 接入目标

通过高德 Web 服务 API 完成门店检索与基础信息补全。

#### 10.1.2 主要接口

- 关键词搜索：`/v3/place/text`
- 详情查询：`/v3/place/detail`

#### 10.1.3 建议请求参数

- `keywords=店名`
- `city=目标城市 adcode`
- `citylimit=true`
- `extensions=all`
- `key=高德 Web 服务 Key`

#### 10.1.4 主要使用字段

- `name`
- `id`
- `location`
- `address`
- `business_area`
- `biz_ext.rating`
- `biz_ext.cost`
- `tag`

#### 10.1.5 字段映射规则

- 评分：使用 `biz_ext.rating`
- 人均价格：使用 `biz_ext.cost`
- 特色菜：使用 `tag` 字段按分隔符切分后取前 3 项
- 地图位置：使用 `location`
- 商圈：使用 `business_area`

#### 10.1.6 接入注意事项

- 高德关键词搜索一次只支持一个关键词，因此批量导入需要服务端并发调用多次。
- 同名门店会返回多个结果，V1 需具备基础候选匹配策略。
- 部分门店字段可能为空，例如评分、人均、特色菜未覆盖，此时前端需容错显示。

### 10.2 OCR 能力

#### 10.2.1 接入目标

识别截图中的店名文本，供后续门店检索使用。

#### 10.2.2 输入规则

- 最多 3 张图片
- 支持连续截图上传
- 单次任务中 OCR 结果统一合并处理

#### 10.2.3 输出要求

- 返回原始文本
- 支持按行结构输出
- 允许 OCR 存在误差，后续由清洗逻辑二次纠正
- 必须配合前端展示处理中状态与超时失败提示

## 11. 匹配与容错策略

### 11.1 匹配策略

- 默认取高德搜索结果第一条作为主匹配结果
- 若结果明显缺字段或命中不准，允许切换候选项
- 优先依赖城市限制提高准确率

### 11.2 容错规则

- 无评分：显示“暂无评分”
- 无人均：显示“暂无人均”
- 无特色菜 / 标签：默认隐藏该字段，不显示占位文案
- 无查询结果：标记为“未匹配成功”，默认不导入
- OCR 识别出脏词：保留原始文本，清洗后再匹配
- OCR 超时：15 秒内未完成则提示失败并允许重试

## 12. 产品规则细化

### 12.1 默认选择规则

- 查询成功的门店默认全选
- 未匹配成功的门店默认不选

### 12.2 导入规则

- 用户点击“确定导入”时，只导入当前选中的门店
- 重复收藏同一 `poi_id` 时，提示已存在，避免重复插入

### 12.3 地图规则

- 常规情况下，地图页默认自动缩放到包含当前筛选结果的所有收藏门店
- 若用户刚完成一次导入，地图页应优先聚焦到本次导入批次所在城市，并默认选中该批次评分最高的门店
- 点击 marker 后展示门店基础信息
- 门店基础信息卡片提供“取消收藏”按钮
- 地图底部面板展开后必须展示当前门店的附近商家列表
- 附近商家列表按距离排序，距离越近越靠前
- 地图页需要支持基于附近商家的距离计算与展示
- “导航到这”应以让用户进入其常用地图导航为目标，不应只局限于微信内查看位置

## 13. 非功能需求

### 13.1 性能

- 识别 10 家门店时，导入预览应在可接受时间内完成
- 高德查询应支持并发，但要控制频率，避免触发限流
- OCR 对 1 到 3 张截图的处理应尽量控制在 8 秒内，避免用户长时间等待

### 13.2 可用性

- 导入流程应尽量短，减少跳转
- 导入前必须可修改勾选结果，避免误收藏
- 预览页第一屏必须优先呈现可导入商家，避免无效结果干扰用户判断
- 地图页在导入完成后的首次进入必须让用户清楚知道“刚刚导入的是哪一批门店”

### 13.3 可测试性与验收

- 地图底部面板展开后，附近商家列表必须有自动化测试覆盖
- 测试必须覆盖：
  - 有附近商家时，展开后能真实渲染列表项
  - 点击附近商家后，当前商家与列表同时更新
  - 导入完成后进入地图页时，默认聚焦逻辑正确
- 若上述用例未通过，不可视为地图页交互完成

### 13.4 数据安全

- 高德 Key 与 OCR 密钥仅存放在服务端
- 小程序端不直接暴露敏感密钥

## 14. 开发建议

### 14.1 建议架构

- 客户端：微信小程序原生开发
- 服务端：轻量 API 服务
- 数据库：SQLite 或 PostgreSQL

### 14.2 原因

- 小程序端适合做导入、预览、地图展示
- 服务端适合封装 OCR、高德查询、字段清洗和入库逻辑
- 后续如需扩展网页后台或更多数据源，服务端架构更容易演进

## 16. 风险与待确认项

### 16.1 风险

- OCR 文本质量不稳定，会影响门店匹配准确率
- 高德部分门店字段可能不完整
- 同名品牌多门店时，默认第一条结果可能不总是正确

### 16.2 当前已确认

- V1 不做评论正文能力
- V1 不做推荐 / 避雷标签
- 地图只做分布展示，不做门店间距离计算
- 截图上传上限为 3 张
- 高德作为 POI 数据源
- 微信小程序原生地图用于展示
